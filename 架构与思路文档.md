
# ARCHITECTURE.md

# Lucid Music — 架构设计与整体思路

> HarmonyOS Stage 模型 · ArkTS · Native C++ JSVM 引擎 · LX Music 音源生态兼容

---

## 一、项目概述

基于鸿蒙 HarmonyOS 的音乐播放器应用，支持内置 5 平台音源搜索播放，以及通过 LX Music 自定义音源脚本实现第三方音源扩展。

**核心原则**：
- **底层逻辑全量保留**（JSVM、5 大平台 API、AudioSourceManager、PlaybackIntentState、AVPlayer）
- **UI 层完全重写**（毛玻璃容器、一镜到底转场、组件化拆分）
- **依赖倒置**（UI 依赖抽象服务，不直接 new 业务类）

## 四、核心数据流

### 4.1 全局状态（`MusicAppState`）
- 存储当前播放歌曲、播放状态、队列
- 由 `AudioPlayerService` 更新
- 被 `MiniPlayerBar`、各页面 UI 组件订阅

### 4.2 播放意图（`PlaybackIntentState`）
- 任意页面触发播放：设置 `pendingPlaySongId` / `remoteUrl` + `requestVersion++`
- `Index.ets` 监听变化 → 分发到 `MediaService` 执行播放
- 播放成功后自动更新 `MusicAppState`

### 4.3 搜索流程
```
首页点击搜索框（一镜到底放大上浮）
  ↓
SearchPage（覆盖层）
  ├── 初始态：显示热门搜索 + 搜索历史
  ├── 输入态：显示联想建议（关键词标红）
  └── 回车/点击建议 → 携带关键词跳转
        ↓
SearchResultPage（覆盖层替换）
  ├── 显示搜索结果列表（关键词延续标红）
  ├── 点击整行 → 触发全局播放
  └── 点击四点 → 原位弹出菜单（下一首播放/稍后播放/添加到歌单）
```

## 五、UI 组件化策略

### 5.1 毛玻璃容器（`AreaWithHdsTabBar`）
- 利用 HdsTabs 的 `barFloatingStyle` 实现沉浸光感
- `TabContent` 留空，只取 `tabBar` 插槽
- 支持 `BarPosition.Start` / `End` 切换位置

### 5.2 底部栏下沉
- `LucidTabBar` 和 `MiniPlayerBar` 由各个页面自行调用
- 每个页面决定显示哪一层：
  - 首页/音乐厅/我的 → 显示 Tab栏 + 迷你条
  - 搜索页/结果页 → 仅显示迷你条
  - 播放页 → 两者都不显示

### 5.3 一镜到底转场
- 共享元素锚点：`SearchInputArea` 组件绑定 `id('search_input')`
- 首页点击搜索框 → 搜索框放大上浮 → 覆盖层 `SearchPage` 显现
- 返回时倒放动画 → 搜索框收缩归位

## 六、扩展性预留

| 预留方向 | 实现方式 |
| :--- | :--- |
| 新增页面 | 在 `features/` 下新建模块，按需调用底部组件 |
| 歌单详情页 | 只需调用 `MiniPlayerBar`，不调用 `LucidTabBar` |
| 播放页全屏 | 不调用任何底部组件 |
| 新歌单操作 | 四点菜单中预留“添加到歌单”入口，可展开二级菜单 |

### 关键目录职责速查表

| 目录 | 职责 | 依赖关系 |
| :--- | :--- | :--- |
| **`entry/pages/Index.ets`** | 根容器，组装 Swiper（3个Tab） + 搜索覆盖层，监听播放意图 | 依赖 `features/home`、`musichall`、`mine`、`search` |
| **`features/*/view/`** | 各页面 UI 布局，组合公共组件，决定底部显示哪些条 | 依赖 `common/components/*` 和 `common/musicbasic` |
| **`features/search/viewmodel/`** | 搜索页和结果页的状态管理、API调用、数据转换 | 依赖 `common/musicbasic`（状态/意图）和 `common/utils`（标红） |
| **`common/components/`** | 纯 UI 组件，无业务逻辑，可独立预览 | 依赖 `common/musicbasic`（仅迷你条读取状态） |
| **`common/musicbasic/`** | 应用的数据核心（歌曲模型、播放状态、意图） | 不依赖任何 UI 层 |
| **`common/constants/`** | 全局固定值（平台枚举、设计规范） | 无依赖 |
| **`common/utils/`** | 无状态纯函数工具（字符串处理、时间格式化） | 无依赖 |

这个架构完全符合你的要求：
1. **底层逻辑全量保留**（`js_audio_engine` + `musicbasic` 原封不动）
2. **UI 组件完全下沉**（每个页面独立调用底部条，搜索页控制覆盖层）
3. **一镜到底锚点明确**（`SearchInputArea` 作为共享组件）
4. **扩展性极强**（新增页面只需复制底部组合代码）?
