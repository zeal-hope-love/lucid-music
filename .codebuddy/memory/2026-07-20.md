## 播放缓存功能（对齐 lx-music-mobile，无限磁盘缓存）
- 需求：在线歌"播过不重取/不重下"。用户明确：**不设上限、不做 LRU 淘汰（无限缓存）**；**缓存整首歌全部内容（歌曲信息+封面+歌词+音频）**；**无启用开关，功能默认直接启用**。
- 实现：
  - 新建 `common/musicbasic/src/main/ets/util/AudioCacheHelper.ets`（单例）：缓存键 `platform_remoteSongId`；目录 `context.cacheDir/audio_cache/`；每首歌落 3 文件 `<key>.mp3`/`.jpg`/`.lrc`，索引 `.index.json`（`AudioCacheEntry{key,audioPath,imagePath,lyricPath,meta:SongMeta,size,lastAccess}`）。`getCachedAudio(key)` 命中且音频文件存在则返条目（O(1) 查表+accessSync 校验）；`cacheRemoteSong(key,songItem,url)` 后台 `request.downloadFile` 下载音频+封面、`fs.writeTextSync` 写歌词、写索引（含 SongMeta 元数据），`Set<string>` 同键去重，失败静默忽略；`clearAll()` 删目录+索引；`getTotalSize()` 扫描目录全部文件，`getCachedSongCount()` 返回索引条目数。已在 `common/musicbasic/Index.ets` barrel 导出 `AudioCacheHelper/SongMeta/AudioCacheEntry`。
  - `AudioPlayerController.playCurrent` 接入：先查 `getCachedAudio`，命中→`loadAndPlay(本地路径)` 并回填 title/singer/coverUrl/lyricText（**不覆盖 songItem.src**，以便缓存被清理后回退在线取流）；未命中→原 `resolveUrlForQueue` 取流，`loadAndPlay(远程url)` 立即播，同时 fire-and-forget `cacheRemoteSong` 落盘。`loadAndPlay` 新增本地绝对路径（`url.startsWith('/')`）分支走 `avPlayer.url`，避免误入 raw fd 分支。
  - 设置页 `features/mine/.../view/CacheSection.ets`（新建）+ `SettingsPage.ets`：在"自定义音源"下方留 28vp 间距加同款标题「缓存」，展示已用缓存大小（`getTotalSize` 格式化 KB/MB/GB）、缓存歌曲数、清理缓存按钮（`clearAll`+toast）。
- 注意：SongItem 无 album/duration 字段，meta.album 留空、meta.duration=0（由 AVPlayer 播放时填 progressMax）；local 歌（无 platform/remoteSongId）不参与缓存。

## 播放缓存「一直是 0」修复（下载环节）
- 现象：在线歌唱完，设置页缓存仍显示 0 KB / 0 首，疑似失效。
- 根因：原 `AudioCacheHelper` 用 `@ohos.request` 的 `request.downloadFile` 下载音频，失败被静默吞掉、且无法拿到真实错误码；多数音乐平台 CDN（网易云/QQ 等）要求 `Referer`/`User-Agent` 头，缺失直接 403，导致下载从未落盘 → 缓存恒为 0。
- 修复（`common/musicbasic/src/main/ets/util/AudioCacheHelper.ets`）：
  - 下载改用 `@kit.NetworkKit` 的 `http` 模块（项目 js_audio_engine 已验证可用），`expectDataType: ARRAY_BUFFER` 写盘，带 connect/read 超时，非 200/空响应显式抛错并打日志（可观测真实原因）。
  - 新增 `refererFor(platform)` 按平台返回 Referer（wy→music.163.com、tx→y.qq.com、mg/kw/kg 对应官网），下载音频+封面时附加；统一带 `User-Agent`。
  - `getCacheDir()` 无 context 时返回空串（原会误落到根目录 `/audio_cache`）；移除无用 `DownloadTaskLike` 接口与 `request` 依赖。
- 重要前提（需告知用户）：缓存**只对在线歌（搜索/歌单里带 platform+remoteSongId 的远程歌曲）生效**；本地导入歌曲本来就是文件，不进缓存，显示 0 属正常。设置页 `CacheSection` 仅在 `aboutToAppear` 刷新，下载进行中打开可能仍显示 0，需等下载完成或重进设置页查看。
- 验证：lint 0 错误；需真机/模拟器播一首在线歌后看设置页是否出现缓存。

## 播放缓存「日志无 AudioCacheHelper、缓存恒 0」真正根因（搜索单曲路径缺 platform/remoteSongId）
- 现象：播了好几首在线歌，缓存仍 0，且日志里**完全没有 `AudioCacheHelper` 输出**（连 cacheRemoteSong 的失败日志都没有）→ 说明 `cacheRemoteSong` 根本没被调用。
- 根因：搜索结果点播走的是 `RemotePlayHelper.playSong` → `PlaybackIntentState.remoteUrl` → `PlaybackIntentDispatcher` 的「远程单曲播放」分支（非 `playCurrent` 的队列取流分支）。该分支 `new SongItem()` 只设了 src/title/singer/cover/lyric，**没设 platform / remoteSongId**；且 `playSong`/`intentState` 也从未传递这两个字段。结果 `playCurrent` 里 `isRemote = !!platform && !!remoteSongId` = false → `cacheKey = ''` → 缓存落盘分支被完全跳过。（缓存只在这三条件齐全时触发，之前的「http 下载 + Referer」修复方向对但没到真正断点。）
- 修复（补齐单曲播放链路的 platform/remoteSongId 传递，3 处）：
  - `common/musicbasic/src/main/ets/model/PlaybackIntentState.ets`：新增 `@Trace remotePlatform` + `@Trace remoteSongId` 字段，`resetRemote()` 一并重置。
  - `common/musicbasic/src/main/ets/util/RemotePlayHelper.ets` 的 `playSong`：设置 `intentState.remotePlatform = item.source`、`intentState.remoteSongId = item.songId`（SearchResultItem.source/songId 均为 string）。
  - `features/player/src/main/ets/service/PlaybackIntentDispatcher.ets`：构建 songItem 时（在 `resetRemote()` **之前**）补 `songItem.platform = intentState.remotePlatform`、`songItem.remoteSongId = intentState.remoteSongId`，并在日志打印二者。
- 效果：此后单曲播放的 songItem 带全 platform+remoteSongId → playCurrent 中 cacheKey 非空 + src 是 http URL → 触发 `cacheRemoteSong` 落盘；下次同曲先命中缓存直接播本地文件（连换源都省）。
- 验证：3 文件 lint 0 错误；需真机播一首搜索来的在线歌，日志应出现 `AudioCacheHelper: cached key=...`，设置页缓存不再为 0。

## AudioCacheHelper mkdir failed code=13900015（EEXIST 中止缓存）
- 现象：日志 `AudioCacheHelper: mkdir failed {"code":13900015}`。13900015 = OpenHarmony EEXIST（目录已存在）。
- 根因：`doCache` 原逻辑 `fs.mkdirSync(dir, true)` 在**目录已存在时抛 EEXIST**，随后 `catch` 中 `return` 直接中止整段缓存落盘。结果：首播（目录不存在）能缓存，第二首起（目录已存在）抛 EEXIST 提前 return → 缓存恒 0。与上一处「缺 platform/remoteSongId」是“放了好几首还是 0”的叠加原因。
- 修复：`AudioCacheHelper.ets` 新增 `ensureCacheDir(dir)` 私有方法——先 `fs.accessSync(dir)` 判断、存在就跳过 mkdir、失败才记日志返回 false；`persistIndex` 与 `doCache` 两处 mkdir 均改用 `ensureCacheDir`。lint 0 错误。
- 经验：OpenHarmony `fs.mkdirSync(path, true)` 在目标目录已存在时会抛 EEXIST，不能裸调后 return；务必先 accessSync 判断或用递归前判空。
