## 用户需求
修复音乐播放器中与"播放列表/播放队列"相关的三类问题：
1. 播放列表（底部弹层与全屏播放页）默认应显示"最近播放"。
2. 在任意歌单内点击歌曲后，播放列表的标题与队列应切换为当前歌单。
3. 全屏详情页（封面/歌名/歌词/弹层列表）显示的内容必须与正在播放的内容严格一致（含远程歌曲）。

## 产品概述
lucid_music 的播放队列存在 V1（`AppStorage 'songList'`）与 V2（`MusicAppState.playQueueIds`/`getCurrentSongItem()`）两套数据源。迷你播放条读 V2（始终正确），而详情页、歌词、播放列表弹层读 V1。由于 V1 仅在切歌/进入详情页时构建、启动时从未初始化、且远程歌曲时 `playQueueIds` 不更新，导致详情页内容与实际播放不一致；同时多处入口把歌单标题写死为"歌单详情"，使"切换为当前歌单"看似未生效。

## 核心功能
- 启动即把 V2 队列同步到 V1，使默认播放列表（最近播放/全部歌曲）在弹层与详情页正确呈现。
- 远程歌曲被建模为"单元素队列"（`playQueueIds=[remoteId]` + `remoteSongItem`），V1/V2 完全一致，详情页不再错位。
- 点击歌单内歌曲时，队列与标题切换为该歌单真实名称（修复写死标题）。
- 本地/远程切换边界正确：切到本地队列时清空 `remoteSongItem`，停留在远程单元素队列时保留。


## 技术栈
- HarmonyOS ArkUI（ArkTS 严格模式），模块：entry（路由入口）、common/musicbasic（模型/组件）、features/player（播放核心）。
- 状态：AppStorageV2（`MusicAppState` 唯一可信源）+ AppStorage V1（`songList`/`selectIndex`/`currentPlaylistTitle` 供 V1 组件消费）。

## 实现方案
核心思路：**统一以 V2 `MusicAppState` 为唯一队列源**，并新增"显示队列"构建逻辑，使 V1 `songList` 始终镜像真实播放内容（含远程）。远程歌曲建模为单元素队列，避免 `playQueueIds` 与正在播放内容脱节。

关键决策：
1. 在 `MusicAppState` 新增 `buildDisplayQueue(): SongItem[]` 与 `syncDisplayQueueToV1()`，集中处理"远程优先"的队列解析，消除 `AudioPlayerController` 与 `PlaybackPage` 中重复的、忽略远程的构建逻辑。
2. `resolveSongAtQueueIndex` 增加回退：当 `remoteSongItem` 存在且 `playQueueIds[i] === remoteSongItem.id` 时返回 `remoteSongItem`，使 V2 解析与 V1 完全一致。
3. 分发器本地分支补充 `appState.remoteSongItem = null`（否则上一段远程播放会污染本地播放）；远程分支写入 `playQueueIds=[remoteId]`、`selectIndex=0` 并保留 `remoteSongItem`。
4. `switchToIndex/playNext/playPrevious` 仅在目标索引解析为本地曲时才清空 `remoteSongItem`，避免远程单元素队列被误清空导致播放中断。
5. 启动与进入详情页均调用 `syncDisplayQueueToV1()`，保证默认与每次出现都同步。
6. 导航标题：用 `PlaylistRow.title`（处理 `string | Resource`）替换写死的"歌单详情"。

## 性能与可靠性
- `buildDisplayQueue` 为 O(n) 线性解析，n 为队列长度（通常数十），开销可忽略；复用现有 `resolveSongAtQueueIndex`，无额外 DB 遍历。
- 远程回退仅在 `remoteSongItem` 非空且 id 匹配时触发，不影响本地路径。
- 不引入新状态字段，仅复用 `playQueueIds`/`remoteSongItem`，保持向后兼容。

## 实现注意事项
- `AppStorage.setOrCreate('songList', ...)` 在 key 已存在时会更新其值，可安全复用；但需保证每次切歌都重新赋值新数组引用以触发 `@StorageLink` 刷新。
- `resolveSongAtQueueIndex` 被 `MiniPlayerBar` 等多处复用，新增回退分支不改变本地行为。
- 改动局部、低风险；构建需在 DevEco Studio 内验证（命令行 hvigorw 依赖 IDE 环境）。

数据流向（修复后）：
```mermaid
flowchart TD
  A[用户点击歌曲/SongListPage] --> B[PlaybackIntentDispatcher]
  B -->|本地| C[appState.playQueueIds=歌单ids, remoteSongItem=null]
  B -->|远程| D[appState.playQueueIds=[remoteId], remoteSongItem=远程曲, selectIndex=0]
  C & D --> E[AudioPlayerController.syncStateForNewSong]
  E --> F[appState.buildDisplayQueue 远程感知]
  F --> G[AppStorage 'songList' / 'selectIndex' / 'currentPlaylistTitle']
  G --> H[详情页/歌词/弹层 读 V1 songList]
  A --> I[MiniPlayerBar 读 V2 getCurrentSongItem]
  H & I --> J[内容一致]
```
