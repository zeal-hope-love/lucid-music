## 用户需求
1. 播放列表默认显示为"最近播放"。
2. 在歌单（本地歌单 / 音乐厅歌单 / 最近播放 / 收藏）内点击歌曲后，播放列表切换为该歌单（队列与标题同步为当前歌单）。
3. 修复播放顺序切换（顺序 / 随机 / 单曲）功能，当前点击无效果。
4. 强调：顺序 / 随机 / 单曲 的播放模式状态管理必须做好，集中到 model 层（`common/musicbasic/src/main/ets/model`）。

## 核心功能
- 应用启动时将播放队列初始化为"最近播放"歌曲；若尚无记录则回退到全部本地歌曲，保证有内容可播。
- 在歌单内点击歌曲时，播放队列切换为该歌单，且播放列表弹层标题显示对应歌单名称。
- 播放列表弹层正确展示完整队列、可点击切歌；顺序 / 随机 / 单曲三种模式真正生效（上一首 / 下一首 / 自动连播遵循当前模式）。
- 播放模式状态以 `MusicAppState` 为唯一可信源，枚举定义收归 model 层，消除控制器重复字段与状态分歧隐患。


## 技术栈与架构
- HarmonyOS ArkUI（ArkTS 严格模式）。沿用现有分层：common/musicbasic（模型、状态、DB、公共组件）、features/player（控制器与播放 UI）、entry（路由与启动入口）。
- 状态管理：以 `MusicAppState`（@ObservedV2，经 AppStorageV2 连接）为播放模式与队列的唯一可信源；AppStorage V1 作为 V1 组件（ControlAreaComponent）的兼容层（@StorageLink）。
- 播放请求统一链路：SongListPage → PlaybackIntentState → Index.ets 监听 requestVersion → dispatchShellPlaybackIntent → AudioPlayerController。
- 模块依赖约束：model 层不可 import player（防循环依赖），故 `MusicPlayMode` 枚举必须定义在 musicbasic/model。

## 根因分析
1. **默认队列错误**：`entry/src/main/ets/pages/Index.ets` 的 `aboutToAppear` 将 `playQueueIds` 初始化为 `getSongList()`（全部本地歌曲），而非"最近播放"。
2. **播放列表弹层损坏（顺序切换"无效果"的真正根因）**：`AudioPlayerController.syncStateForNewSong` 与 `PlaybackPage.syncCurrentSongToAppStorage` 把 `songList` 强制写成 `[当前歌曲]`、`selectIndex` 强制置 0。导致全屏播放器的 `ControlAreaComponent.musicListBuilder` 永远只渲染 1 首，点击调用 `switchToIndex(0)` 恒为当前曲（无效）。对照 `MiniPlayerBar.playlistSheetBuilder` 已正确使用 `appState.playQueueIds`，证明队列逻辑本身可用，仅全屏弹层数据绑定损坏。
3. **播放模式状态管理分散（用户强调项）**：`MusicPlayMode` 枚举定义在 `features/player/constants/PlayerConstants.ets`（player 模块），model 层只能以裸 `number` 持有 `playMode`；`AudioPlayerController` 又维护一份 `private playMode` 私有字段，构造函数未从 `appState` 同步，存在状态分歧隐患。
4. **缺少当前歌单标题跟踪**：`MusicAppState` 无"当前歌单标题"字段，弹层标题写死，无法满足"切换为当前歌单"展示需求。

## 实现方案

### 1. 播放模式状态集中到 model 层（核心改进）
- 新增 `common/musicbasic/src/main/ets/model/PlayMode.ets`：定义 `MusicPlayMode` 枚举（SINGLE_CYCLE=0, ORDER=1, RANDOM=2）与切换顺序常量 `PLAY_MODE_CYCLE_ORDER = [ORDER, RANDOM, SINGLE_CYCLE]`。这是枚举唯一可信源，musicbasic 与 player 均可引用。
- `MusicAppState` 改造为强类型唯一可信源：
  - `playMode: MusicPlayMode = MusicPlayMode.ORDER`（替代裸 number）。
  - 新增 `setPlayMode(mode: MusicPlayMode): void` → 设 `this.playMode` 并 `AppStorage.setOrCreate('playMode', mode)`（供 V1 ControlAreaComponent 的 @StorageLink 同步）。
  - 新增 `cyclePlayMode(): MusicPlayMode` → 按 `PLAY_MODE_CYCLE_ORDER` 推进并调用 `setPlayMode`，返回新模式（集中切换逻辑，UI 与 AVSession 回调统一走这里）。
- `AudioPlayerController` 去除重复 `private playMode` 字段：`playNext/playPrevious` 改用 `this.appState.playMode`；`setPlayMode(mode)` 改为 `this.appState.setPlayMode(mode)` + `this.avSessionController.setPlayModeToAVSession(mode)`。构造函数不再需要同步（appState 即唯一源，永不失配）。
- `PlayerConstants.ets`：删除 `MusicPlayMode` 枚举；`PlayModeSwitchList` 改为 `export const PlayModeSwitchList = PLAY_MODE_CYCLE_ORDER`（别名，保持既有消费者不变）。
- `features/player/Index.ets`：`MusicPlayMode` / `PLAY_MODE_CYCLE_ORDER` 改为从 `musicbasic` 再导出（保持对外兼容）。
- `AVSessionController.ets` 与 `ControlAreaComponent.ets` 的 `MusicPlayMode` 导入改为从 `musicbasic` 取；`ControlAreaComponent.switchPlayMode()` 改为调用 `this.appState.cyclePlayMode()`（经由控制器或直接调用皆可，模式推进逻辑统一在 model）。

### 2. 修复 songList 强制单元素（核心修复）
在 `AudioPlayerController.syncStateForNewSong` 中，将"只写当前曲"改为"同步完整队列"：遍历 `playQueueIds` 经 `resolveSongAtQueueIndex` 解析，写入 `AppStorage('songList')`，并 `AppStorage.setOrCreate('selectIndex', this.appState.selectIndex)`。`PlaybackPage.syncCurrentSongToAppStorage` 做同样改造（删除 `selectIndex` 强制 0）。`PlayerInfoComponent` / `MusicInfoComponent` / `LyricsComponent` 通过 `songList[selectIndex]` 取当前曲的逻辑保持不变，且弹层获得完整队列。

### 3. 启动默认队列改为"最近播放"
`Index.ets` 的 `aboutToAppear`：取 `getRecentPlaySongs().filter(s => s.id > 0)`（过滤负 ID 远程歌曲），非空则 `playQueueIds = 映射 id` 且 `currentPlaylistTitle = '最近播放'`；否则回退 `getSongList()` 映射 id 且 `currentPlaylistTitle = '全部歌曲'`。同步 `AppStorage('currentPlaylistTitle')`。

### 4. 串联歌单标题与切歌播放
- `SongListPage.onSongTap` / `onRfSongTap`：发起播放前写入 `this.intentState.pendingPlaylistTitle = this.param.title`（远程歌单用 `this.playlistInfo.name`）。
- `PlaybackIntentDispatcher`：消费 `pendingQueueIds` 后设置 `appState.currentPlaylistTitle = intentState.pendingPlaylistTitle || appState.currentPlaylistTitle` 并 `AppStorage.setOrCreate('currentPlaylistTitle', ...)`，清空 `pendingPlaylistTitle`。
- `PlaybackIntentDispatcher` 的 `pendingPlaySongId` 分支：命中后真正切歌播放——置 `selectIndex`、`remoteSongItem = null`、`syncStateForNewSong()`、`isPlay = true`、`playCurrent()`，使 MiniPlayerBar 弹层点击生效。

### 5. 弹层展示真实队列与歌单标题
- `ControlAreaComponent`：新增 `@StorageLink('currentPlaylistTitle') currentPlaylistTitle: string = '最近播放'`；`playListTitle()` 标题改为展示 `this.currentPlaylistTitle`（附 `this.songList.length` 首数）。`musicListBuilder` 已用 `songList` + `switchToIndex(index)`，队列修复后即生效。
- `MiniPlayerBar.playlistSheetBuilder`：标题 `Text('播放列表')` 改为 `Text(this.appState.currentPlaylistTitle)`。

## 性能与边界
- `songList` 每次切歌 O(n) 重建（n 为队列长度，通常数十~数百），开销可忽略；PlayerInfoComponent 等本就会随 selectIndex 重渲染，无额外成本。
- 远程歌曲（负 ID）不在本地 store，`resolveSongAtQueueIndex` 返回 undefined，启动与队列构建时自然跳过；远程歌单走 RemotePlayHelper 既有流程，本次不改动其队列行为。
- 空队列时：MiniPlayerBar 已有"播放列表为空"空态；ControlAreaComponent 弹层补充空态判断，避免空白。
- 向后兼容：不删除/重命名既有 AppStorage 键，仅新增 `currentPlaylistTitle`；`songList` 语义由"当前曲"变为"完整队列"，所有消费者均按 `songList[selectIndex]` 取当前曲，行为一致。

## 关键代码结构
```ts
// common/musicbasic/src/main/ets/model/PlayMode.ets
export enum MusicPlayMode {
  SINGLE_CYCLE = 0,
  ORDER = 1,
  RANDOM = 2
}
/** 模式切换顺序：顺序 → 随机 → 单曲循环 */
export const PLAY_MODE_CYCLE_ORDER: MusicPlayMode[] = [
  MusicPlayMode.ORDER,
  MusicPlayMode.RANDOM,
  MusicPlayMode.SINGLE_CYCLE
]

// MusicAppState 新增/改造
@Trace public playMode: MusicPlayMode = MusicPlayMode.ORDER
@Trace public currentPlaylistTitle: string = '最近播放'

public setPlayMode(mode: MusicPlayMode): void {
  this.playMode = mode
  AppStorage.setOrCreate('playMode', mode)
}

public cyclePlayMode(): MusicPlayMode {
  const order = PLAY_MODE_CYCLE_ORDER
  const idx = order.indexOf(this.playMode)
  const next = order[(idx + 1) % order.length]
  this.setPlayMode(next)
  return next
}
```
